# 输出与引用规则：数据跨境合规

## 何时读取本文件

当准备输出出海数据流动风险雷达、转交建议、正式报告或任何带引用的结论时，必须先读取本文件。

## 本文件影响什么决策

本文件决定聊天摘要与正式 Markdown 报告结构、引用格式、确定性标签、转交建议和用户可见表达边界。

## 不适用场景

仅做数据流识别、来源边界判断或检索规划时，不以本文件替代方法边界和搜索策略。

> 本文件供 `legato-cross-border-data-flow-compliance` 运行时使用。
> 与 `method-and-source-boundaries.md` 协同工作：来源边界和材料准入规则见该文件，本文件聚焦引用格式、确定性标签和输出纪律。
> 本 Skill 独立打包，不依赖 `_shared` 或上级目录。

## 1. 输出层级

Legato 产品中，本 Skill 输出固定为"聊天摘要 + 强制正式 Markdown 交付物"：

1. 聊天窗口简短 Markdown 摘要，只帮助用户快速理解方向。
2. 正式 Markdown 报告交付物，是研究任务的强制交付面。

Markdown / Word 导出由产品交付物功能承担。

内部工具协议：

- 首轮研究任务必须使用 `write_file` 生成完整 Markdown 交付物，再使用 `present_artifacts` 展示。
- 同一研究的后续修改、补充、纠错、扩写或结构调整，必须使用 `edit_file` 更新既有交付物，再使用 `present_artifacts` 展示。
- 后续对话如果是无关新研究，必须重新使用 `write_file` 创建新的 Markdown 交付物，再使用 `present_artifacts` 展示。
- 纯解释既有交付物中某段含义、且不新增结论、依据、结构或行动清单时，可以只在聊天窗口回答。
- 工具名只用于 Skill 内部执行协议、eval 和研发说明，不得进入用户可见报告。

## 2. 聊天窗口回答

聊天摘要应像律师给客户提供建议——结论先行，用通俗语言说，语气自然。先让用户知道"这个数据流动场景的主要风险是什么"，再讲最关键的理由或建议。

底线要求：
- 必须包含结论方向（"主要风险在 XX 侧""整体可控""需要重点关注 XX"等）。
- 必须说明已生成正式报告。
- 不得把完整报告内容塞进聊天消息。
- 不得暴露内部工具名、MCP ID 或供应商名称。

## 3. 正式 Markdown 报告

首轮研究任务必须生成一份最终 Markdown 报告。正式报告结构以 `assets/data-flow-compliance-report-template.md` 为唯一真源。本文件只规定输出层级、引用、确定性标签、转交建议和用户可见表达边界，不再单独维护报告章节。

生成或更新正式报告前必须先读取 `assets/data-flow-compliance-report-template.md`，并按模板中的标题、章节、表格和条件触发段落填充内容。不得用本文件的聊天摘要结构替代正式报告，也不得恢复旧版的“检索范围与来源说明”“引用与证据”“结论限制与人工复核触发”等内部说明式章节。

## 4. 来源层级

### 4.1 中国侧来源层级

| 层级 | 来源 | 使用规则 |
| --- | --- | --- |
| 高 | 《个人信息保护法》《数据安全法》《网络安全法》原文；GB/T 35273 等国家标准；网信办正式通知、办法、指南；工信部、公安部等主管机关正式文件 | 可支撑决定性结论 |
| 中 | 权威法律指南（如 Chambers 数据保护章节）、高质量律所深度文章、专家材料、可定位的合规指南 | 可辅助框架和实务判断 |
| 低 | 媒体快讯、行业资讯、非正式评论、微信公众号转述 | 只作动态信号 |
| 待确认 | 未返回可定位片段、版本不明、来源冲突或事实缺失 | 必须提示人工复核 |

### 4.2 欧盟侧来源层级

| 层级 | 来源 | 使用规则 |
| --- | --- | --- |
| 高 | GDPR 条文原文（Art.44-49）；EDPB 正式指南、建议和约束性决定；DPA 正式裁决和执法决定；CJEU 判例（C-311/18 等）；EU SCCs 官方文本 | 可支撑决定性结论 |
| 中 | ICLG、Chambers、Legal500 数据保护指南；权威律所深度文章；IAPP 专业报告 | 可辅助框架和实务判断 |
| 低 | 行业新闻、博客评论、社交媒体讨论 | 只作动态信号 |
| 待确认 | 未返回可定位片段、版本不明、来源冲突或事实缺失 | 必须提示人工复核 |

### 4.3 用户可见来源命名

报告、聊天回答、引用脚注和来源说明只使用用户能理解的来源名称：

| 用户可见名称 | 使用场景 |
| --- | --- |
| 本次挂载资料 / 项目资料 | 用户本轮选择或挂载的项目文件、内部材料 |
| 中国法律法规数据库 | flk.npc.gov.cn 等法规库 |
| 主管机关官方来源 | 网信办、工信部、公安部等主管机关页面 |
| 欧盟官方法规库 | EUR-Lex、EDPB 官方文件 |
| 高质量法律指南 | ICLG、Chambers、Legal500、DLA Piper Data Protection 等 |
| 专业实务解读 | 律所、会计师事务所、专业机构文章 |
| 联网公开来源 | 公开网页、新闻、专业文章 |

不要在用户可见内容中写供应商名称、内部配置编号或工具函数名。

## 5. 脚注格式

优先使用 Markdown 脚注：

```markdown
正文中的关键法律结论[^1]。

[^1]: [来源标题](URL) - 层级 | 来源类型 | 日期 | 结论说明
```

中国法条脚注示例：

```markdown
[^1]: [个人信息保护法 第三十八条](https://flk.npc.gov.cn/) - 高 | 法规原文 | 2021-11-01 | 规定个人信息出境应满足的条件之一
```

欧盟来源脚注示例：

```markdown
[^2]: [EDPB Recommendations 01/2020](https://edpb.europa.eu/system/files/2021-06/edpb_recommendations_202001vo.2.0_supplementarymeasurestransferstools_en.pdf) - 高 | 官方指南 | 2021-06 | 提供 TIA 六步框架和补充措施
```

指南类脚注示例：

```markdown
[^3]: [ICLG Data Protection - China](https://iclg.com/practice-areas/data-protection-laws-and-regulations/china) - 中 | 专业指南 | 2024 | 用于搭建中国数据出境路径框架；关键法律结论仍需官方来源核验
```

## 6. 确定性标签

| 标签 | 含义 | 适用来源 |
| --- | --- | --- |
| [法规原文] | 直接引用法条或法规原文 | 《个人信息保护法》《数据安全法》GDPR 条文、国家标准 |
| [权威解读] | 来自主管机关官方解读或 EDPB 正式指南 | 网信办官方 FAQ、EDPB 指南、DPA 裁决 |
| [一般评论] | 来自律所文章、行业分析、指南概述 | ICLG、Chambers、律所 client alert |
| [待验证] | 来源不确定、无法确认版本、或 AI 生成补全 | 未找到官方核验的二手信息 |

## 7. 精度纪律

- 源文件说 "may require" 不得转述为 "requires"。
- 源文件说 "in general" 或 "typically" 不得转述为 "always"。
- 源文件说 "should" 不得转述为 "must"。
- GDPR 条文中的 "shall" 和 "may" 有严格法律含义差异：shall = 强制义务，may = 许可或裁量。
- 多个来源说法不一致时，分别列明，不揉成折中结论。
- 中国法中"应当"和"可以"同样具有严格的义务等级差异。

## 8. 必须引用的内容

- 所有决定性法律结论（包括中国侧和欧盟侧）。
- 数据出境路径判断（安全评估、标准合同、认证、豁免）。
- 条件、门槛、例外、处罚、备案、审批、期限。
- GDPR 传输机制判断（adequacy、SCC、BCR、例外）。
- TIA 初筛结论和风险等级。
- "当前有效""最新变化""草案 / 已生效"等时效判断。
- 主管机关、办理路径、材料要求、官方入口。
- 多法域比较中的关键差异。
- 依赖二手来源或弱来源的判断。
- 实务操作建议中的关键步骤。

## 9. 不得引用的内容

- 只有文件名、知识库名、网页标题，但没有可定位原文片段。
- 搜索摘要不足以支撑结论的内容。
- 推测性常识补全。
- AI 生成答案本身。
- 路由建议本身（如"建议转 Skill 2"不需要法律引用）。

### 引用真实性底线

以下内容属于 LLM 模板占位符或幻觉产物，不得出现在任何引用、脚注或来源标注中：

- `[Valid]`、`[Effective]`、`[Current Date]`、`[现行有效]`、`[有效]`
- `[Body]`、`[X]`、`[Name]`、`[Date]`、`[数量]`、`[客户名称]`
- 未经实际检索验证的 URL 或域名（不得编造看起来合理但未实际访问的来源地址）
- 无对应脚注实体的纯数字标记（如 `[1]`、`[2]` 但文末无对应脚注条目）

引用必须来自实际检索命中或知识库召回的结果。不确定来源是否真实存在时，标为 [待验证] 而非伪造引用。

## 10. 冲突处理

### 10.1 来源内部冲突

来源冲突时：
- 分别列明冲突来源。
- 比较效力层级、发布时间、适用范围和具体事实。
- 如果仍无法收口，标为"不明确"或"需当地确认"。

不要把冲突来源揉成一个看似确定的折中结论。

### 10.2 中国法 vs GDPR 冲突场景

当中国数据出境规则与 GDPR 跨境传输规则存在张力时：

- **分别分析**：先按中国侧规则分析出境路径，再按 GDPR 规则分析传输机制，不混合讨论。
- **冲突标注**：如果两条路径的义务存在重叠或矛盾（如安全评估结果与 TIA 结论不一致），明确标注冲突点。
- **不替代判断**：本 Skill 不做"中国法 vs GDPR 优先级"的最终判断，只输出两套规则的各自要求和冲突识别。
- **触发人工复核**：涉及两套规则同时适用的场景，必须在结论限制中提示需要人工综合判断。

## 11. 翻译和原文

- 对欧盟法域，关键术语尽量保留英文原文（如 Standard Contractual Clauses、Transfer Impact Assessment）。
- 对中国法域，关键术语保留中文原文（如个人信息保护法、数据出境安全评估）。
- 用中文交付时，第一次出现原文术语后应解释含义。
- 不把机器翻译或非官方译文当作无风险原文。
- GDPR 条文引用必须注明引用的是官方英文版还是中文译本，中文译本不得作为权威依据。
