# 投资标的筛查 — 输出与引用规则

> 总则：所有法律结论和风险定性必须有可定位来源。无可定位片段不得挂引用，搜索摘要不得直接当法律结论。
> 总则：聊天摘要是帮助用户快速判断方向的简短回复，不能替代正式 Markdown 交付物。正式交付物是本 Skill 的唯一权威输出。

## 何时读取本文件

- 准备输出聊天摘要、正式报告、引用脚注或法律红旗结论前。
- MCP 未返回字段、返回空值、数据冲突或需要表达“未披露”时。

## 本文件影响什么决策

- 决定公开数据如何引用、哪些字段不能补全、风险结论强度和正式报告结构。

## 不适用场景

- 不用于选择 MCP 工具；工具路由见 `mcp-routing.md`。

## 1. 输出层级

固定为"聊天摘要 + 强制正式 Markdown 交付物"。

内部工具协议：

- 首轮必须 `write_file` 生成完整 Markdown 交付物，再 `present_artifacts` 展示。
- 同一研究的后续修改，必须 `edit_file` 更新既有交付物，再 `present_artifacts` 展示。
- 无关新研究，必须 `write_file` 创建新交付物。
- 纯解释既有交付物内容且不新增结论时，可以只聊天回答。
- 工具名只用于 Skill 内部执行协议、eval 和研发说明，不得进入用户可见报告。

## 2. 聊天窗口回答

聊天摘要应像律师给客户提供建议——结论先行，用通俗语言说，语气自然。先让用户知道"这家公司能不能投、主要风险在哪"，再讲最关键的发现。

底线要求：
- 必须包含结论方向和红绿灯状态（"绿灯/黄灯/红灯"）。
- 必须说明已生成正式报告。
- 不得把完整报告内容塞进聊天消息。
- 不得暴露内部工具名、MCP ID 或供应商名称。

聊天摘要是方向性提示，不是正式交付物。用户应以正式 Markdown 报告为准。

## 3. 数据纪律

MCP 数据只写工具返回的内容：

- 列表型字段全量展示，不省略中间项。
- 统计数字必须从原始数据计算，不凭印象写。
- MCP 未返回的字段标注"未披露"，不编造、不补全。
- 不写"已处置""已闭环"等 MCP 未显式返回的结论性状态。
- 处罚日期、机关、文号必须直引 MCP 原文。

## 4. 来源层级

| 层级 | 来源 | 使用规则 |
| --- | --- | --- |
| 高 | 企业工商登记数据、失信被执行人记录、行政处罚公告、法院裁判文书、法规原文 | 可支撑风险判断 |
| 中 | 知产数据、融资记录、招聘信息、荣誉资质、权威法律解读、高质量指南 | 辅助判断，不单独支撑结论 |
| 低 | 联网新闻、行业资讯、非正式评论 | 只作动态信号 |
| 待确认 | MCP 返回空或不可用、来源冲突或版本不明 | 标注"未获取"或"待确认"，不编造 |

## 5. 脚注格式

优先使用 Markdown 脚注：

```markdown
正文中的关键结论[^1]。

[^1]: [来源名称](URL) - 高 | 工商登记数据 | 数据日期 YYYY-MM-DD | 说明该来源支撑的具体结论
```

如果来源为企业工商与风险数据（MCP 查询）：

```markdown
[^1]: 企业工商与风险数据 - 高 | 失信被执行人记录 | 2026-05-25 | 显示该公司存在未履行的失信被执行记录
```

如果来源为联网搜索：

```markdown
[^1]: [信用中国行政处罚公示](URL) - 中 | 行政处罚公告 | 2026-05-25 | 该来源显示该公司于 2026 年受到行政处罚
```

如果来源为法律解读或分析：

```markdown
[^1]: [专业法律解读](URL) - 中 | 法律解读 | 2026 | 用于理解法律框架；风险定性仍需法规原文核验
```

## 6. 确定性标签

报告中的每条风险判断和法律结论必须标注确定性等级：

| 标签 | 含义 | 适用来源 |
| --- | --- | --- |
| [法规原文] | 直接引用法条或法规原文 | 高层级：官方法规库、政府公报、法律条文 |
| [权威解读] | 来自权威法律指南、专业机构、主管机关说明 | 中层级：高质量法律解读 |
| [一般评论] | 来自联网新闻、行业分析、非正式评论 | 低层级：媒体、行业资讯 |
| [待验证] | MCP 数据缺失、来源不确定、或未找到官方核验 | 待确认层级 |

使用规则：

- 红旗信号的风险定性必须标注确定性标签。
- MCP 数据本身标注来源类型（如"工商登记数据""行政处罚公告"），不需要 [法规原文] 标签。
- 法律影响分析（如"股权冻结可能导致交割障碍"）需标注 [法规原文] 或 [权威解读]。
- 不得以 [一般评论] 或 [待验证] 作为唯一支撑给出确定性风险判断。

## 7. 精度纪律

### 语义精度

- MCP 返回"存续"不得转述为"经营状况良好"。
- MCP 返回有失信记录不得转述为"信用崩塌"（除非有多项叠加证据）。
- MCP 返回有行政处罚不得省略处罚内容直接定性为"重大合规风险"。
- MCP 未返回的字段不得用"无"替代，应标注"未披露"。
- 多个数据来源说法不一致时，分别列明，不揉成折中结论。

### 风险定性精度

- 风险定性必须区分"MCP 数据信号"与"法律影响判断"。
- 不得将"有被执行记录"直接等同于"信用崩塌"——需结合金额、次数、时间综合判断。
- 不得将"有行政处罚"直接等同于"合规存疑"——需评估处罚内容、严重程度和行业影响。
- 不得将"知产出质"直接等同于"资产虚高"——需结合出质比例和企业融资需求判断。

### 联网搜索精度

- 联网搜索结果只作动态信号，不得替代 MCP 结构化数据。
- 搜索摘要不得直接写成风险结论。
- 搜到的新闻或行业文章需标注"联网公开来源"和具体日期。

## 8. 必须引用的内容

- 红旗信号及其风险定性。
- 行政处罚的日期、机关、文号和处罚内容。
- 失信被执行人记录的来源和状态。
- 股权冻结、经营异常等特殊状态的法律影响分析。
- MCP 数据驱动的风险判断（引用数据返回内容）。
- 联网搜索补充的风险信号（引用具体来源）。
- 依赖二手来源或弱来源的判断。

## 9. 不得引用的内容

- 只有文件名、知识库名、网页标题，但没有可定位原文片段。
- 搜索摘要不足以支撑风险结论的内容。
- 推测性补全（如"该行业通常存在此类风险"）。
- AI 生成答案本身。
- MCP 工具名称或内部配置编号。

## 10. 用户可见来源命名

| 内部来源 | 用户可见名称 |
| --- | --- |
| `qcc-company` / `qcc-risk` / `qcc-executive` / `qcc-ipr` / `qcc-operation` / `qcc-history` | 企业工商与风险数据 |
| `yuandian-law` | 中国法律法规库 |
| 运行时可见联网搜索工具 | 联网公开来源 |
| `query_legato_ragflow` | 本次挂载资料 / 项目资料 |

用户可见输出中不得出现企查查、元典、MCP 等供应商名称或工具函数名。

## 11. 冲突处理

来源冲突时：

- 分别列明冲突来源及其确定性标签。
- 比较来源类型（MCP 数据 > 联网搜索 > 一般评论）、数据日期和覆盖范围。
- 如果仍无法收口，标为"不明确"或"需人工复核"。

不要把冲突来源揉成一个看似确定的折中结论。

MCP 数据与联网搜索冲突时：

- 以 MCP 结构化数据为准。
- 如联网搜索提供 MCP 未覆盖的维度，分别列明。

## 12. 翻译和原文

- 公司名称、法律术语保留原文。
- 用中文交付时，英文术语首次出现后应解释含义。
- 不把机器翻译当作无风险原文。
