# 输出与引用规则

## 何时读取本文件

- 准备输出聊天摘要、正式 Markdown 报告或引用脚注前。
- 发现来源冲突、无可定位片段、案例和法规支撑强度不一致时。

## 本文件影响什么决策

- 决定哪些结论可以挂引用、哪些必须降级为待验证，以及报告中如何表达来源层级。

## 不适用场景

- 不用于选择法规、案例或企业数据检索路径；检索路由见 `mcp-routing.md`。

## 0. 总则：所有法律结论必须有来源

本 Skill 输出的每一条法律结论——包括法条引用、门槛数值、处罚标准、程序要求、案例裁判口径——必须有可定位来源支撑。

- 有来源的结论：标注确定性标签 + 脚注引用。
- 无来源的结论：标为 [待验证]，并明确告知用户"未找到可定位来源，建议人工核验"。
- 禁止将搜索摘要、AI 推断或模型训练知识直接包装为法律结论。
- 禁止仅给出法规名称而不注明具体条款和条文内容。

违反上述规则的输出视为无效输出，必须回溯修正。

## 1. 输出层级

Legato 产品中，本 Skill 输出固定为“聊天摘要 + 强制正式 Markdown 交付物”：

1. 聊天窗口简短 Markdown 摘要，只帮助用户快速理解方向。
2. 正式 Markdown 报告交付物，是研究任务的强制交付面。

Markdown / Word 导出由产品交付物功能承担。

内部工具协议：

- 首轮研究任务必须使用 `write_file` 生成完整 Markdown 交付物，再使用 `present_artifacts` 展示。
- 同一研究的后续修改、补充、纠错、扩写或结构调整，必须使用 `edit_file` 更新既有交付物，再使用 `present_artifacts` 展示。
- 后续对话如果是无关新研究，必须重新使用 `write_file` 创建新的 Markdown 交付物，再使用 `present_artifacts` 展示。
- 纯解释既有交付物中某段含义、且不新增结论、依据、结构或行动清单时，可以只在聊天窗口回答。
- 工具名只用于 Skill 内部执行协议、eval 和研发说明，不得进入用户可见报告。

## 2. 聊天窗口回答

聊天窗口回答要短，帮助用户快速判断结果方向。

默认结构：

```markdown
## 简要结论
- 

## 关键依据 / 风险
- 

## 正式报告
- 已生成正式 Markdown 交付物，并已展示。

## 限制与复核
- 
```

要求：

- 先给答案，再讲依据。
- 只放关键结论和限制。
- 不把完整研究报告塞进聊天消息。

## 3. 正式 Markdown 报告

首轮研究任务必须生成一份最终 Markdown 报告，并通过产品交付物能力展示；法条定位、案例定位、简单法规检索和较窄问题也不例外。Skill 不固定底层输出路径或文件名模板。

报告结构：

```markdown
# {中国法研究报告标题}

## 1. 任务范围
## 2. 执行摘要
## 3. 输入事实与当前假设
## 4. 检索范围与来源说明
## 5. 中国法任务分型与分析框架
## 6. 法律依据
## 7. 案例支撑
## 8. 核心问题分析
## 9. 风险分级 / 结论强度
## 10. 建议行动与分工
## 11. 引用与证据
## 12. 结论限制与人工复核触发
```

## 4. 来源层级

| 层级 | 来源 | 使用规则 |
| --- | --- | --- |
| 高 | 法条原文、规范性文件原文、司法解释、权威案例原文、官方监管材料 | 可支撑决定性结论 |
| 中 | 高质量律所、咨询机构、学术或专业解读 | 可辅助理解，不单独支撑决定性结论 |
| 低 | 媒体快讯、行业资讯、非正式评论 | 只作动态信号 |
| 待确认 | 工具未返回可定位片段或事实缺失 | 必须提示人工复核 |

## 4.1 确定性标签体系

报告中的每条法律结论必须标注确定性标签：

| 标签 | 含义 | 适用来源 |
| --- | --- | --- |
| [法规原文] | 直接引用法条、法规、司法解释或规范性文件原文 | 高层级来源：法律法规库原文、政府公报 |
| [权威解读] | 来自最高法/最高检指导性案例、权威案例裁判要旨、官方释义或高质量专业解读 | 高层级案例、官方释义、权威指南 |
| [一般评论] | 来自律所文章、咨询机构分析、学术论文、行业解读 | 中层级来源 |
| [待验证] | 来自搜索摘要、AI 生成内容、无法确认来源或未找到官方核验 | 低层级或无来源 |

标签规则：

- 标签描述来源性质，不描述结论可信度。即使结论看起来合理，也不得仅因"看起来正确"就提升标签层级。
- 同一结论只能标注一个确定性标签，以最高可用来源为准。
- 标签必须紧跟在结论句之后或脚注中使用，不得遗漏。

## 4.2 精度纪律

本 Skill 严格遵守以下精度纪律：

1. **措辞保真**：源文件说 "may require" 不得转述为 "requires"；"typically" 不得转述为 "always"；"原则上" 不得转述为 "必须"。
2. **程度保真**：源文件说 "部分案例" 不得转述为 "普遍做法"；"可以" 不得转述为 "应当"。
3. **状态保真**：源文件说 "征求意见稿" 不得省略"征求意见"阶段；"已废止" 不得继续引用为现行有效。
4. **引用完整性**：不得仅引用法条号而不给出条文内容；不得仅给出法规名称而不注明条款。
5. **搜索摘要不是法律结论**：联网搜索返回的摘要片段是线索，不是法律依据。将其转化为结论前必须回溯到原文确认。
6. **来源缺失即标注**：无法找到可定位来源的结论，必须标为 [待验证]，不得隐藏来源缺失的事实。
7. **伪引用过滤**：输出前过滤 `[Valid]`、`[Current Date]`、`[X]`、`[有效]`、`[现行有效]`、无脚注实体的纯数字标记等占位符。

## 5. 用户可见来源命名

报告、聊天回答、引用脚注和来源说明只使用用户能理解的来源名称：

| 用户可见名称 | 使用场景 |
| --- | --- |
| 中国法律法规库 | 法律法规、规范性文件、司法解释、法条原文、时效状态 |
| 中国案例库 | 权威案例、典型案例、普通案例、裁判口径 |
| 企业法律事实数据 | 企业涉诉、处罚、执行等法律事实补充 |
| 本次挂载资料 / 项目资料 | 用户本轮选择或挂载的项目文件、内部材料、研究包资料 |
| 联网公开来源 | 官方页面、监管动态、公开专业文章和网页材料 |

不要在用户可见内容中写供应商名称、内部配置编号或工具函数名。

## 6. 脚注格式

优先使用 Markdown 脚注，必须包含日期字段：

```markdown
正文中的关键法律结论[^1]。

[^1]: [来源标题](URL) - 高 | 法条原文 | 2026-05-25 | 说明该来源支撑的具体结论
```

脚注字段说明：

- 字段 1：来源标题（链接或来源名称）
- 字段 2：来源层级（高 / 中 / 低 / 待确认）
- 字段 3：来源类型（法规原文 / 权威解读 / 一般评论 / 待验证）
- 字段 4：日期（精确到日或年均可，但必须填写）
- 字段 5：该来源支撑的具体结论说明

中国法法律结论应尽量包含法规 / 规则名、条号或章节、原文片段或 URL、日期和支撑点；不得只写“参见相关规定”。
禁止只有域名级标签的引用，例如 `[samr.gov.cn]`、`[csrc.gov.cn]`、`[gov.cn]` 或 `[domain]`。如果只取得域名、网页标题或搜索摘要而没有具体页面、法规 / 规则名称、条号 / 章节或可定位片段，应降级为 [待验证]，不得挂引用编号。

如果中国法律法规库返回了可定位原文片段但没有 URL：

```markdown
[^1]: 中国法律法规库返回片段 - 高 | 法条原文 | 2026-05-25 | 返回结果包含法规名称、条号和原文片段；外部链接待产品返回结构支持
```

如果当前只取得实务解读但尚未取得法条原文：

```markdown
[^1]: [律所实务解读标题](URL) - 中 | 一般评论 | 2026 | 用于理解实务操作；关键法律结论仍需法条原文核验
```

## 7. 必须引用的内容

- 决定性法律结论。
- 条件、门槛、例外、处罚、备案、审批、期限。
- “当前有效”“最新变化”“草案 / 已生效”等时效判断。
- 案例裁判口径、法院观点、案号、裁判日期。
- 依赖二手来源或弱来源的判断。
- 实务操作建议中的关键步骤。

## 8. 不得引用的内容

- 只有文件名、知识库名、网页标题，但没有可定位原文片段。
- 工具返回摘要不足以支撑结论的内容。
- 推测性常识补全。
- `[Valid]`、`[Current Date]`、`[X]`、`[有效]`、`[现行有效]`、无脚注实体的纯数字标记等伪引用。

## 9. 冲突处理

来源冲突时：

- 分别列明冲突来源。
- 比较效力层级、发布时间、适用范围和具体事实。
- 如果仍无法收口，标为“不明确”或“需人工复核”。

不要把冲突来源揉成一个看似确定的折中结论。

## 10. 策略路径对比表

复杂中国法路径选择问题可增加策略路径对比表：

| 路径 | 适用情形 | 可行性 | 法律依据 | 关键风险 | 证据强度 | 建议动作 |
| --- | --- | --- | --- | --- | --- | --- |

触发条件：

- 用户问题涉及“怎么做”“选择哪种方案”“是否通过替代路径实现”“是否申报 / 备案 / 调整交易结构”等。
- 至少存在两条可行路径。
- 不同路径在审批、备案、处罚风险、交易成本、时间或责任分配上有实质差异。

不触发：

- 法条定位、案例定位、定义解释或单点规则确认。
- 只有一个合法路径。
- 缺少来源支撑的路径不得写成确定推荐，只能标为待验证。
