# 输出与引用规则

## 何时读取本文件

准备输出聊天摘要或正式 Markdown 报告时，必须读取本文件。

## 本文件影响什么决策

- 决定聊天摘要和正式报告的输出结构。
- 决定引用格式、来源层级和确定性标签。
- 决定来源覆盖状态和结论强度的表达。
- 决定"参考来源与引用资料"章节的写法。
- 决定无可定位来源时是否允许挂引用编号。

## 不适用场景

- 不用于法律推理方法论（见 `legal-reasoning-methods.md`）。
- 不用于诉讼策略方案的整体设计（见 `dispute-resolution-methods.md`）。

---

## 1. 输出层级

本 Skill 输出固定为"聊天摘要 + 正式 Markdown 交付物"：

1. **聊天窗口简短摘要**：帮助用户快速理解方向。
2. **正式 Markdown 报告**：强制交付面，不是可选项。

---

## 2. 聊天窗口回答

聊天摘要应像律师给客户提供建议——结论先行，用通俗语言说，语气自然。先让用户知道"这个问题的答案是什么"，再讲最关键的理由、风险或建议方向。

诉前合同争议预防场景中，聊天摘要必须先回答"应如何设计争议解决路径和条款"，而不是默认从"诉讼能否支持"开场。若满足 B2B 合同可预先设计、核心痛点是举证责任/证据规则分配、隐蔽事实难以鉴定或对方配合不确定等条件，应主动说明"建议优先约定仲裁"，再简要说明条款应如何写、还需保留哪些反证和复核点。

底线要求：
- 必须包含结论方向（"能打""有风险""关键看条款怎么写"等）。
- 诉前合同设计场景必须包含推荐路径（如仲裁优先/诉讼备选）和条款机制方向。
- 必须说明已生成正式报告。
- 不得把完整策略报告塞进聊天消息。
- 不得暴露内部工具名、MCP ID 或供应商名称。

条款建议：当回答涉及合同条款设计时，应给出示例性条文，不只说"建议优化表述"。如果用户已经说明第三方鉴定成本高、效果弱或对方配合不确定，不要把第三方鉴定作为默认首选；应优先提出事实推定、对方反证、联合确认/逾期不配合后果和仲裁条款的组合方案。

---

## 3. 正式报告

正式报告结构以 `assets/dispute-resolution-report-template.md` 为唯一真源。

> **与报告模板的分工**：本文件只提供引用、来源层级和精度规则，不定义报告章节结构。报告结构的唯一真源是 `assets/dispute-resolution-report-template.md`。生成正式报告前必须先读取该模板，并按任务分型选择诉前合同设计结构或起诉/应诉策略结构。

---

## 4. 来源层级

| 层级 | 来源 | 使用规则 |
|---|---|---|
| 高 | 法条原文、司法解释、指导性案例、权威案例原文 | 可支撑决定性结论 |
| 中 | 高质量律所文章、咨询机构分析、学术解读 | 可辅助理解，不单独支撑决定性结论 |
| 低 | 媒体快讯、行业资讯 | 只作背景参考 |
| 待确认 | 无可定位片段或事实缺失 | 必须提示人工复核 |

---

## 5. 来源覆盖状态与结论强度

正式报告的执行摘要必须包含"结论强度 + 来源覆盖状态"。两者不是一回事：

- **结论强度**：基于法律依据、事实基础、证据链、对方抗辩和实务路径综合判断，为强 / 中等 / 弱 / 需进一步论证。
- **来源覆盖状态**：基于本次可定位来源质量判断，为强 / 中 / 弱 / 未覆盖。

| 覆盖状态 | 条件 | 允许输出 | 必须提示 |
|---|---|---|---|
| 强 | 有法规原文、司法解释、法院/仲裁机构规则、权威案例或用户挂载资料的可定位片段 | 可以给出有依据的结论 | 标明主要依据、日期和支撑点 |
| 中 | 有高质量律所/机构实务文章或专业指南，但官方来源不足 | 可以给出初步判断 | 标注基于二手来源，建议律师或官方来源复核 |
| 弱 | 只有零散文章、新闻、搜索摘要或不可稳定访问来源 | 只能给方向性风险提示 | 明确覆盖有限，不建议直接执行 |
| 未覆盖 | 未找到可定位来源 | 不挂引用编号，不伪装成已核验结论 | 说明未找到权威来源和补料路径 |

---

## 6. 确定性标签

报告中的每条法律结论必须标注确定性标签：

| 标签 | 含义 |
|---|---|
| [法规原文] | 直接引用法条、法规或司法解释原文 |
| [权威解读] | 来自指导性案例、权威案例裁判要旨或官方释义 |
| [一般评论] | 来自律所文章、咨询机构分析或学术论文 |
| [待验证] | 来自搜索摘要、AI 生成内容或未找到官方核验 |

---

## 7. 正文脚注与"引用/参考来源"章节

本 Skill 的正式报告**必须包含独立的"引用/参考来源"章节**（报告最后一章），并满足两层要求：

1. 正文关键法律结论使用 Markdown 脚注 `[^N]`。
2. 章节内列出每个脚注实体，并可附来源清单表格。

格式：

```markdown
## 引用/参考来源

[^1]: 《中华人民共和国民法典》第 617 条 - 高 | 法规原文 | 2021-01-01 | 支撑出卖人质量瑕疵担保责任。
[^2]: {仲裁规则标题及条款} - 高/中 | 仲裁规则 | {日期} | 支撑当事人可约定证据规则或仲裁路径。

### 来源清单

| # | 来源 | 层级 | 类型 | 日期 | 定位信息 | 支撑的结论 |
|---|---|---|---|---|---|---|
| 1 | 《中华人民共和国民法典》第 617 条 | 高 | 法规原文 | 2021-01-01 | 第 617 条 | 出卖人质量瑕疵担保责任 |
| ... | | | | | |
```

要求：
- 每条法律结论、仲裁路径判断、条款效力判断、举证责任判断和关键实务规则都必须有脚注支撑。
- 列出报告正文中每一条脚注对应的完整来源信息，脚注不能只出现在表格里。
- 按来源层级从高到低排列。
- 引用最低字段：法规/规则/案例/文章标题、条号/章节/段落/可定位片段、来源标题或机构、URL/数据库定位/用户挂载资料片段、日期、支撑点。
- 没有可定位来源的结论标为 [待验证] 或"未找到可定位来源"，不得伪造引用，不得挂 `[^N]`。
- 过滤 `[Valid]`、`[Current Date]`、`[有效]` 等伪引用。

---

## 8. 脚注格式

优先使用 Markdown 脚注：

```markdown
正文中的关键法律结论[^1]。

[^1]: [来源标题](URL) - 高 | 法条原文 | 2026-06-06 | 说明该来源支撑的具体结论
```

如来源来自用户挂载资料，脚注格式为：

```markdown
[^2]: 本次挂载资料《{文件名}》{页码/章节/片段定位} - 高/中 | 项目资料 | {资料日期或未注明日期} | 说明该资料支撑的事实结论
```

---

## 9. 精度纪律

1. **措辞保真**：源文件说"可以"不得转述为"应当"；"部分案例"不得转述为"普遍做法"。
2. **来源缺失即标注**：找不到可定位来源的结论，标为 [待验证]，不挂引用编号。
3. **禁止绝对化**：法律结论不写"一定""必然""肯定"，写"较大可能""存在风险""需进一步确认"。
4. **伪引用过滤**：输出前过滤所有模板占位符和无脚注实体的 `[1]`、`[2]`。
5. **禁止域名级引用**：不得只写域名、文件名、知识库名或网页标题而无可定位片段。

---

## 10. 冲突处理

来源冲突时：
- 分别列明冲突来源。
- 比较效力层级、发布时间和适用范围。
- 无法收口时标为"不明确"或"需人工复核"。

---

## 11. 人工复核触发

涉及以下情况时，必须在报告中提示人工复核：
- 诉讼胜负的关键判断。
- 重大金额的风险评估。
- 证据薄弱或关键事实不明确。
- 法律依据存在解释争议。
- 对方抗辩模拟中存在无法排除的重大风险。
- 仲裁条款有效性、可仲裁性、仲裁机构规则、格式条款风险、双方议价地位或行业惯例会影响结论。
