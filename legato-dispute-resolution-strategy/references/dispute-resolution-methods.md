# 争议解决方法论

## 何时读取本文件

开始任何争议解决策略任务时，必须首先读取本文件。本文件确定任务分型（诉前预防/原告/被告）、对应的分析策略和整个任务的执行框架。

## 本文件影响什么决策

- 确定诉前预防/原告/被告三类模式及对应的分析切入点。
- 确定质证意见的撰写方法。
- 确定争议解决策略方案的输出结构。
- 确定本任务的检索策略和停止条件。

## 不适用场景

- 不用于法律推理方法论（见 `legal-reasoning-methods.md`）。
- 不用于诉讼程序导航（见 `litigation-procedure-guide.md`）。
- 不用于证据三性审查和证明力评估（见 `evidence-assessment.md`）。
- 不用于模拟对方抗辩（见 `opponent-simulation.md`）。

## 1. 任务分型与角色判断

先根据用户描述判断任务分型，再判断我方在案件中的角色：

| 用户表述 | 判断为 | 分析模式 |
|---|---|---|
| "想在合同中约定""未来诉讼能否支持""举证责任怎么分配""证据规则能不能约定""仲裁还是诉讼更好" | 诉前合同争议预防 | 预防型（Preventer） |
| "对方欠我钱不还""我方起诉""我方主张""对方违约" | 原告 | 建设型（Builder） |
| "对方起诉我""收到应诉通知""被起诉了""收到传票" | 被告 | 拆弹型（Breaker） |
| 无法判断 | 默认原告 | 列出需补充的信息 |

信息不足时以"起诉方"视角分析，但在报告中明确标注"当前假设为起诉方视角，如为应诉方，分析框架和策略重点将有所不同"。如果问题明显发生在合同签署或修订前，优先进入诉前合同争议预防模式，而不是默认进入原告模式。

## 2. 诉前合同争议预防模式

### 适用条件

同时满足以下多数条件时，进入预防型分析：

1. 交易仍处在合同起草、谈判、续签、补充协议或条款可修改阶段。
2. 用户关注的是未来争议中谁来证明、证明到什么程度、对方不配合怎么办。
3. 争议事实具有隐蔽性、技术性、专业性或发生后难以直接还原。
4. 第三方鉴定、司法鉴定或现场排查成本高，且对方配合不确定。
5. 用户希望通过合同约定事实推定、举证责任分配、证据规则、故障确认程序、仲裁/诉讼路径来降低未来争议风险。

不满足上述条件、只是要求审查普通质量条款/付款条款/违约条款时，不应进入本 Skill，应转向合同审查或中国法研究能力。

### 思维起点

预防型分析不是问"将来法院会不会百分百支持"，而是问：

1. 未来争议中最难证明的事实是什么。
2. 这个事实能否通过合同约定转化为更容易证明的基础事实。
3. 哪种争议解决路径更能承接当事人对证据规则和举证责任的安排。
4. 条款怎样写才能既降低我方举证风险，又保留对方合理反证空间，避免被认定为格式条款、显失公平或排除对方主要权利。

### 仲裁优先触发规则

在国内民商事 B2B 合同中，如果核心痛点是举证责任、证据规则或隐蔽技术事实分配，且合同尚可预先设计，应主动评估并优先建议仲裁。理由是仲裁通常更适合承接当事人意思自治，尤其是双方对证据规则、事实推定、举证责任、专家评审或行业技术问题的预先安排。

但仲裁建议不得绝对化，必须同时提示以下复核点：

- 仲裁条款必须明确、有效，写清仲裁机构和仲裁事项。
- 争议事项须具有可仲裁性。
- 仲裁机构规则是否承认或尊重当事人对证据规则的特别约定。
- 条款是否构成不合理格式条款、显失公平或排除对方主要抗辩权。
- 仲裁裁决执行和保全便利性是否符合交易需要。

### 输出重点

预防型分析必须输出：

1. 用户原始问题和事实结构：交易关系、未来故障/违约场景、最难证明的事实。
2. 核心策略判断：是否建议通过合同约定转移或重构举证风险。
3. 路径选择：优先比较仲裁、诉讼、调解；符合仲裁优先规则时，将仲裁列为首选。
4. 条款设计方案：给出可落地的示例条文，至少包含基础事实、推定效果、对方反证、故障/证据确认流程。
5. 未来攻防：供应商/相对方可能攻击条款效力、定位方法、格式条款风险时，我方如何应对。
6. 证据与流程机制：保留实物、通知期限、联合确认、逾期不配合后果、内部质量记录等。

### 避免过度程序化

在预防型场景中，不要默认把"第三方鉴定"设计成唯一或最高优先方案。若用户已经说明第三方检测难以证明隐蔽质量问题，应把第三方检测定位为辅助机制或兜底机制，优先通过合同事实推定、供应商参与确认/逾期不参与后果、反证责任和仲裁路径来降低举证风险。

## 3. 原告/被告双模式

### 原告模式（建设型）

**做什么**：帮我方把主张建起来。

1. **找请求权基础**：按合同→不当得利→侵权的顺序逐一检视，找到最适合的请求权。
2. **拆构成要件**：把请求权拆成几个必须满足的条件（如合同违约要拆成：合同成立→我方履约→对方违约→造成损失→因果关系）。
3. **逐项对照事实**：每个要件有没有对应的事实支撑？事实有没有证据？
4. **评估证据链**：哪些要件证据强、哪些弱、哪些缺。
5. **预判对方攻击**：对方最可能从哪里下手（质疑合同效力？质疑损失金额？主张不可抗力？）。

**输出重点**：请求权基础选择、构成要件完成度、证据补强方向、对方可能攻击点及我方应对。

### 被告模式（拆弹型）

**做什么**：帮我方把对方的主张拆掉。

1. **逐项核对对方主张**：对方说的每个事实，哪些确认、哪些否认、哪些不知情。
2. **找程序性抗辩**：诉讼时效是否届满？管辖是否有异议？主体是否适格？
3. **找实体性抗辩**：合同是否无效/可撤销？是否有不可抗力/情势变更？是否可以抵销？
4. **评估反诉可能性**：我方是否有反诉请求权？
5. **攻击对方证据链**：对方证据有没有三性问题？证明力够不够？

**输出重点**：抗辩策略选择、对方证据漏洞、程序性抗辩空间、反诉评估。

## 4. 质证方法论

在评估证据和模拟对方质证时，使用以下方法。

### 四性分析法

对每份证据逐项审查：

1. **真实性**：有没有原件？签名/盖章是否真实？复印件能否与原件核对？
2. **合法性**：取证方式是否合法？是否侵犯隐私？是否属于非法证据排除范围？
3. **关联性**：跟本案待证事实有没有关系？能否证明对方要证明的内容？
4. **证明力**：能不能单独证明待证事实？是否需要其他证据补强？孤证是否足够？

### 策略性评价（四条原则）

1. **攻防一体**：质证不是只挑对方的错，也要保护自己的证据链。
2. **借力打力**：对方拿来证明 A 的证据，如果同时能证明 B（我方主张），就把它变成我方的论据。
3. **预先设防**：预判对方可能怎么攻击我方证据，提前准备好反驳口径。
4. **回扣核心**：每份证据质证完，都要回到我方核心主张上。

## 5. 争议解决策略方案结构

输出策略方案时，先根据任务分型选择结构。

### 诉前合同争议预防结构

1. **用户问题、事实与法律关系**：原始问题、交易关系、未来争议触发场景、最难证明的事实。
2. **核心策略判断**：是否应通过合同机制重构举证风险，是否建议优先约定仲裁。
3. **路径选择**：仲裁、诉讼、调解的适配性、成本、执行和风险。
4. **条款设计方案**：示例条文、条款结构、反证空间、程序保障。
5. **未来攻防模拟**：对方可能挑战条款效力、事实定位和举证分配的路径。
6. **证据与流程机制**：通知、封存、联合确认、逾期不配合、内部记录、检测兜底。
7. **风险与人工复核**：格式条款、议价地位、机构规则、可仲裁性和执行风险。

### 起诉/应诉策略结构

输出起诉/应诉策略方案时，按以下六个部分组织：

### 第一部分：案件概述

当事人、案由、管辖、案件阶段、争议金额、我方角色。

### 第二部分：事实分析

有利事实、不利事实、待核实事实、事实缺口（哪些关键事实还不清楚）。

### 第三部分：法律分析

请求权/抗辩权基础、构成要件逐项对照、法条效力确认、类案裁判口径。

### 第四部分：诉讼/应诉策略

核心主张、证据组织方案（哪些证据先出示、哪些后出示、哪些不主动出示）、攻防要点、Plan B。

### 第五部分：风险评估

主要风险点、发生概率、影响程度、应对措施。

### 第六部分：行动清单

按优先级排列的下一步动作，包括需要补充的证据、需要确认的事实、需要检索的法律问题。

## 6. 证据缺口与补强建议

当证据评估发现缺口时，在报告中说明：

1. **缺什么**：哪个要件缺乏证据支撑。
2. **为什么重要**：这个证据对诉讼请求的支持力度有多大。
3. **可能的获取途径**（简要提示，不展开）：例如"可通过调取银行流水补充""可通过申请法院调查取证获取""可通过证人证言补强"。

证据补强建议不是核心功能，只做简要提示，不做详细的取证指导。诉前预防型场景中，补强建议应转化为合同内的证据流程机制，而不是只列未来诉讼取证动作。

## 7. 检索策略

本 Skill 在分析过程中需要检索法规和案例作为策略支撑。检索不是本 Skill 的主任务，不能把输出做成泛法律研究报告；每一次检索都必须服务于路径选择、请求权/抗辩权、证据风险、仲裁/诉讼方案或条款机制设计。

### 争点到检索任务

不得把用户整段案情直接作为一个泛化 query。先将争议焦点拆成可检索任务，每个任务只解决一个问题：

- 请求权/抗辩权争点：合同违约、质量责任、侵权、不当得利、免责或抗辩事由。
- 举证责任争点：谁证明、证明到什么程度、能否通过合同重构证明风险。
- 证据规则/鉴定争点：隐蔽瑕疵、技术事实、鉴定困难、证据保全、事实推定。
- 路径争点：仲裁条款效力、管辖约定、仲裁/诉讼对证据安排的承接能力。
- 损失和责任范围争点：损失计算、违约金调整、间接损失、赔偿限额。

### 检索顺序

1. **先看项目资料**：如果用户挂载了相关文件（合同、往来函件、案件材料），先从中提取事实。
2. **判断法条优先还是类案优先**：不要把"法条→案例"作为固定顺序。
3. **查中国法律法规库**：用于确认请求权/抗辩权、构成要件、举证责任一般规则、仲裁条款效力、格式条款效力等规范基础。
4. **查中国案例库**：用于确认类似事实结构下法院/仲裁实践的裁判口径，尤其是隐蔽事实、鉴定困难、事实推定、合同举证责任约定是否被采纳。
5. **联网补充**：对最新法规、司法解释、仲裁机构规则、监管动态或公开权威材料进行联网核验。

### 法条优先 / 类案优先判断

| 场景 | 主路径 | 目的 |
|---|---|---|
| 请求权基础、抗辩权基础、构成要件、诉讼时效、管辖/仲裁条款效力 | 法条优先 | 确认规范依据和决定性规则 |
| 举证责任一般规则、格式条款、显失公平、排除对方主要权利 | 法条 + 司法解释优先，必要时补案例 | 判断条款机制的合法边界 |
| 隐蔽瑕疵、技术事实、第三方鉴定困难、对方不配合 | 类案优先，必要时补法条 | 判断裁判者如何处理证明困难和事实推定 |
| 合同中预设证据规则、故障确认流程、逾期不配合后果 | 类案 + 仲裁规则/机构规则 | 判断约定被采纳的实践空间 |
| 诉前预防型场景符合仲裁优先条件 | 仲裁规则 + 类案/实务来源 | 判断仲裁是否更适合承接当事人意思自治 |

### query 构造方法

- **关键词检索**：已有明确制度词、案由、条款类型或裁判关键词时使用，例如"举证责任约定""隐蔽瑕疵""质量缺陷""事实推定""仲裁条款效力""格式条款 举证责任"。
- **语义检索**：事实结构复杂、关键词可能噪音过大或需要找相似事实时使用，句式为"当事人关系 + 关键行为 + 争议焦点 + 后果"。
- **一争点一 query**：每个 query 只承载一个争点，不把请求权、举证责任、仲裁路径、鉴定困难、损失计算混在同一次检索中。
- **诉前预防型 query 示例**：不要检索"供应商零部件质量问题举证责任仲裁诉讼第三方鉴定格式条款"，应拆为"采购合同隐蔽质量缺陷举证责任分配""合同约定事实推定举证责任转移裁判口径""仲裁中当事人约定证据规则举证责任""第三方鉴定困难 产品质量缺陷 责任认定"。

### 检索修正规则

首轮检索无有效来源时，最多进行三轮修正：

1. 关键词无结果时，改用事实结构语义检索；语义检索噪音大时，收窄为制度关键词。
2. 多个争点混杂时，拆成更小的单争点 query。
3. 类案缺少相似事实时，改查上位规则、司法解释、指导/典型案例或仲裁规则，并在报告中降低结论强度。

### 检索目标

- 请求权/抗辩权的法律依据（法条原文 + 效力状态）。
- 构成要件的司法解释和裁判标准。
- 类似案件的裁判口径（胜诉/败诉的关键因素）。
- 特殊案由的专属法律规则。
- 预防型场景中，补充检索仲裁规则、证据规则特别约定、格式条款效力、合同事实推定或举证责任约定相关裁判/规则。

检索结果必须转化为策略结论：对我方有利的规则、对方可能攻击的点、法院路径风险、仲裁路径优势与限制、合同条款应如何调整。不得只罗列法条、案例标题或来源清单。

### 来源层级

| 层级 | 来源 | 使用规则 |
|---|---|---|
| 高 | 法条原文、司法解释、指导性案例 | 可支撑决定性结论 |
| 中 | 高质量律所文章、学术解读 | 辅助理解，不单独支撑决定性结论 |
| 低 | 媒体、行业资讯 | 只作背景参考 |

## 8. 停止条件

**可以停止**：
- 起诉/应诉场景：已完成五步分析（事实提取→焦点识别→证据评估→对手模拟→策略方案）。
- 诉前预防场景：已完成事实结构→举证风险识别→路径选择→条款设计→未来攻防→证据机制。
- 已找到决定性法律依据和类案裁判口径。
- 已识别所有主要风险点并给出应对建议。

**不能停止**：
- 只有案情描述，还没有做要件拆解。
- 还没有评估已有证据的证明力。
- 还没有模拟对方可能的抗辩。
- 诉前预防场景中，未比较仲裁/诉讼路径，或只说"建议第三方鉴定"而没有条款机制。
- 只给了法律依据，没有给策略建议。

---

## 9. 禁止路径

- 不帮写起诉状、答辩状、代理词等法律文书。
- 不做泛合同审查；只处理与争议解决路径、举证责任、证据规则、责任分配、未来攻防直接相关的合同机制。
- 不做排期管理、期限监控、账单计算。
- 不做和解条款谈判模拟。
- 不做刑事、行政诉讼策略。
- 不做跨境争议。
- 不调用 `ask_user_question`。
- 不自行选择知识库、文件夹或文件范围。
- 不在用户可见报告中暴露内部工具名、MCP ID 或供应商名称。
