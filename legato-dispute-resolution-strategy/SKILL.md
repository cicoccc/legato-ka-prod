# 争议解决策略 Skill

> 版本：v0.1
> 日期：2026-06-08
> 状态：draft
> 运行模式：v0.1 固定 `agent`

## 定位

本 Skill 是 Legato 知识助手的中国民事商事争议解决策略能力。它不是查法条、查案例的法律研究工具，而是用争议解决实务方法论指导模型如何拆解事实、评估证据、模拟对手、选择争议解决路径、制定策略。

与 `legato-china-legal-research` 的区别：后者回答"法律是什么"（法条、案例、合规路径），本 Skill 回答"争议怎么布置和怎么打"（合同里如何降低未来举证风险、证据够不够、对方会怎么说、我方该怎么应对）。

## 适用场景

- 起诉方的诉讼策略准备：请求权选择、证据评估、攻防要点。
- 应诉方的抗辩策略分析：抗辩路径、证据漏洞、反诉评估。
- 诉前合同争议预防：围绕举证责任、证据规则、争议解决条款、责任分配设计合同机制。
- 证据完备性评估：已有证据能不能支撑诉讼请求、缺口在哪。
- 对方抗辩模拟：对方可能怎么说、怎么质证我方证据。
- 争议解决方案：诉讼/仲裁/调解的路径选择和风险评估。

## 不适用边界

- 查法条、查案例、确认法律依据，应使用中国法规案例研究 Skill。
- 泛合同审查、商业条款优化、履约管理条款设计；本 Skill 只处理与争议解决路径、举证责任、证据规则、责任承担和未来攻防直接相关的条款。
- 写起诉状、答辩状、代理词、质证意见等法律文书。
- 案件排期管理、期限监控、诉讼费计算。
- 和解条款谈判模拟。
- 跨境争议（本期聚焦国内）。
- 刑事诉讼、行政诉讼。

## 最低输入

必有：
- 案情描述（发生了什么事）。
- 起诉/应诉场景：我方主张/诉讼请求（起诉方），或对方诉请/起诉状内容（应诉方）。
- 诉前合同争议预防场景：拟约定的条款目标或条款草案，以及希望降低的未来争议风险。

建议有（提高分析精度）：
- 现有证据清单或关键证据说明。
- 合同文本或主要条款。
- 争议金额。
- 案件所处阶段（诉前/已起诉/已开庭等）。
- 管辖法院或仲裁机构。
- 是否为 B2B 合同、合同是否仍可修改、双方议价地位、是否已有仲裁条款或可约定仲裁。

信息不足时不调用追问工具；基于已有信息推进，在报告中列出假设、置信度和需要补充的信息。

## 任务分型与角色判断

先判断任务分型，再判断我方角色：

- 用户描述"想在合同中约定""将来诉讼能否支持""举证责任怎么分配""仲裁还是诉讼更好"且合同尚可预先设计 → 诉前合同争议预防，采用预防型分析。
- 用户描述"我方起诉""对方欠我""我方主张" → 原告，采用建设型分析。
- 用户描述"对方起诉我""收到应诉通知""被起诉了" → 被告，采用拆弹型分析。
- 无法判断 → 默认原告，标注"如为应诉方，分析框架将有所不同"。

## 研究执行顺序

1. 判断用户问题是否属于争议解决策略（而非纯法律研究、泛合同审查或文书生成）。
2. 从用户问题中提取原始问题、案情描述、交易阶段和任务分型，确认最低输入是否满足。
3. 读取 `references/dispute-resolution-methods.md`，确认诉前预防/原告/被告三类模式和策略方案结构。
4. 读取 `references/legal-reasoning-methods.md`，确定法律解释方法、请求权/抗辩权检视顺序。
5. 起诉/应诉场景读取 `references/litigation-procedure-guide.md`，确认诉讼程序阶段、特殊案由适配和关键期限；诉前合同争议预防场景只在需要比较未来诉讼风险时读取。
6. 诉前合同争议预防场景先做路径选择（仲裁/诉讼/调解）和条款机制设计；起诉/应诉场景从案情描述中提取法律事实，将诉讼请求或抗辩分解为构成要件清单。
7. 识别争议焦点后，按 `references/dispute-resolution-methods.md` §7 将争点拆成法规、案例、仲裁规则或公开来源检索任务；检索结果必须转化为策略结论，不得只罗列来源。
8. 读取 `references/evidence-assessment.md` 执行证据评估或未来证据机制设计。
9. 读取 `references/opponent-simulation.md`，模拟对方抗辩理由、质证意见或未来条款效力攻击。
10. 综合前九步分析，制定争议解决策略方案；如满足诉前仲裁优先条件，应主动建议约定仲裁并说明限制。
11. 读取 `references/output-and-citation-rules.md` 和 `assets/dispute-resolution-report-template.md`，按任务分型生成正式报告。

## 运行约束

- `mode = agent`。
- 不调用 `ask_user_question`。
- 不使用旧 `query_kb`。
- 不自行选择知识库、文件夹或文件范围；检索范围由前端挂载和研究包配置决定。
- 内部工具名只允许出现在运行配置、研发排查或 eval 中，不进入用户可见报告。
- 首轮研究任务必须使用 `write_file` 生成完整 Markdown 交付物，并使用 `present_artifacts` 展示。
- 同一研究的后续修改，必须使用 `edit_file` 更新既有交付物，再使用 `present_artifacts` 展示。
- 后续对话如果是无关新研究，必须重新使用 `write_file` 创建新的 Markdown 交付物。

## 输出要求

聊天窗口只放简短摘要，包含结论、关键依据、风险和交付物状态；聊天摘要不能替代正式交付物。诉前合同争议预防场景必须先说明建议路径和条款设计方向，不能只回答"诉讼能否支持"。

正式报告必须先读取并使用 `assets/dispute-resolution-report-template.md`。报告必须包含独立的"引用/参考来源"章节，正文关键法律结论必须使用 `[^N]` 角注，脚注实体必须写清来源标题、层级、类型、日期、定位信息和支撑的具体结论。没有可定位来源的判断标为 [待验证]，不挂引用编号。

## 人工复核触发

涉及以下情况时，必须在报告中提示人工复核：
- 诉讼胜负的关键判断。
- 重大金额的风险评估。
- 证据薄弱或关键事实不明确。
- 法律依据存在解释争议。
- 对方抗辩模拟中存在无法排除的重大风险。
- 仲裁条款有效性、可仲裁性、格式条款风险、双方议价地位或具体仲裁机构规则会影响核心结论。
