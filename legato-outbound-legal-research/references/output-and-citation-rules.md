# 输出与引用规则

## 何时读取本文件

- 准备输出聊天摘要、正式 Markdown 报告、引用脚注或来源说明前。
- 检索结果来自官方、指南、律所文章、新闻或一般评论且结论强度需要分级时。

## 本文件影响什么决策

- 决定哪些来源可支撑决定性结论、哪些只能作为框架参考，以及如何在报告中表达限制。

## 不适用场景

- 不用于选择法域来源或构造搜索 query；前者见 `source-routing.md`，后者见 `search-templates.md`。

## 1. 输出层级

Legato 产品中，本 Skill 输出固定为“聊天摘要 + 强制正式 Markdown 交付物”：

1. 聊天窗口简短 Markdown 摘要，只帮助用户快速理解方向。
2. 正式 Markdown 报告交付物，是研究任务的强制交付面。

Markdown / Word 导出由产品交付物功能承担。

内部工具协议：

- 首轮研究任务必须使用 `write_file` 生成完整 Markdown 交付物，再使用 `present_artifacts` 展示。
- 同一研究的后续修改、补充、纠错、扩写或结构调整，必须使用 `edit_file` 更新既有交付物，再使用 `present_artifacts` 展示。
- 后续对话如果是无关新研究，必须重新使用 `write_file` 创建新的 Markdown 交付物，再使用 `present_artifacts` 展示。
- 纯解释既有交付物中某段含义、且不新增结论、依据、结构或行动清单时，可以只在聊天窗口回答。
- 工具名只用于 Skill 内部执行协议、eval 和研发说明，不得进入用户可见报告。

## 2. 聊天窗口回答

聊天摘要应像律师给客户提供建议——结论先行，用通俗语言说，语气自然。先让用户知道"这个问题的答案是什么"，再讲最关键的理由或风险。

底线要求：
- 必须包含结论方向（"可以""不可以""需要满足以下条件"等）。
- 必须说明已生成正式报告。
- 不得把完整报告内容塞进聊天消息。
- 不得暴露内部工具名、MCP ID 或供应商名称。

## 3. 正式 Markdown 报告

首轮研究任务必须生成一份最终 Markdown 报告，并通过产品交付物能力展示；单点流程检索、单法域概览、较窄问题和多法域比较都不例外。Skill 不固定底层输出路径或文件名模板。

正式报告结构以 `assets/outbound-legal-report-template.md` 为唯一真源。本文件只规定输出层级、引用、来源层级、覆盖状态和复核规则，不再单独维护报告章节。

生成或更新正式报告前必须先读取 `assets/outbound-legal-report-template.md`，并按模板中的标题、章节、表格和条件触发段落填充内容。不得用本文件的聊天摘要结构替代正式报告，也不得恢复旧版的“检索范围与来源说明”“出海任务分型与分析框架”“引用与证据”等内部说明式章节。

## 4. 用户可见来源命名

报告、聊天回答、引用脚注和来源说明只使用用户能理解的来源名称：

| 用户可见名称 | 使用场景 |
| --- | --- |
| 本次挂载资料 / 项目资料 | 用户本轮选择或挂载的项目文件、内部材料、研究包资料 |
| 本 Skill 自带法域来源路由表 | 各法域官方来源、机构入口和高质量法律指南的域名范围 |
| 各国官方来源 | 官方法规库、监管机构、法院、仲裁机构、主管机关页面、条约官网 |
| 高质量法律指南 | ICLG、Chambers、Legal500、Lex Mundi、Lexology 等公开指南 |
| 专业实务解读 | 律所、会计师事务所、专业机构文章 |
| 联网公开来源 | 公开网页、监管动态、新闻、专业文章 |
| 企业工商与风险数据 | 企业基础信息、股权、经营、风险、董监高、知产等企业画像 |
| 正式报告交付物 | 最终 Markdown 报告 |

不要在用户可见内容中写供应商名称、内部配置编号或工具函数名。

## 5. 来源层级

| 层级 | 来源 | 使用规则 |
| --- | --- | --- |
| 高 | 法条原文、正式法规数据库、政府公报、监管机构、法院、仲裁机构、官方 FAQ、条约官网 | 可支撑决定性结论 |
| 中 | 高质量法律指南、强实务文章、专家材料、本次挂载资料中的可定位片段 | 可辅助框架、实务和事实判断 |
| 低 | 媒体快讯、行业资讯、非正式评论 | 只作动态信号 |
| 待确认 | 未返回可定位片段、版本不明、来源冲突或事实缺失 | 必须提示人工复核 |

## 5.1 覆盖状态

正式报告应说明每个关键法域或核心问题的覆盖状态：

| 状态 | 含义 | 用户可见表达 |
| --- | --- | --- |
| 强 | 有官方原文、监管 / 法院来源或法规库，且有可定位片段 | “本项有官方或高强度来源支撑” |
| 中 | 主要来自高质量指南或律所实务，官方来源不足 | “本项主要基于二手专业来源，需官方核验” |
| 弱 | 只有开放法源、新闻、中文解读或不可完整访问来源 | “本项覆盖有限，仅作初筛方向” |
| 未覆盖 | 未找到可定位来源 | “未找到可定位来源，不形成确定法律结论” |

未覆盖事项不得挂引用编号，不得使用域名标签伪装为来源。

## 6. 脚注格式

优先使用 Markdown 脚注：

```markdown
正文中的关键法律结论[^1]。

[^1]: [来源标题](URL) - 高 | 官方说明 | 2026-05-25 | 说明该来源支撑的具体结论
```

如果当前只取得官方说明但尚未取得更高层级法条原文：

```markdown
[^1]: [主管机关官方说明](URL) - 高 | 官方说明 | 2026-05-25 | 当前已取得办理页面；尚未取得更高层级法条原文，结论强度相应保留
```

如果来源为高质量法律指南：

```markdown
[^1]: [高质量法律指南标题](URL) - 中 | 专业指南 | 2026 | 用于搭建问题框架；关键法律结论仍需官方来源核验
```

禁止只写域名级引用，例如 `[sfc.hk]`、`[zan.kz]` 或 `[domain]`。引用必须指向具体页面、文件、法规 / 规则名称、条号或章节、日期和支撑点；如果工具只返回域名或标题而无可定位片段，应降级为待确认来源。

## 7. 必须引用的内容

- 决定性法律结论。
- 条件、门槛、例外、处罚、备案、审批、期限。
- “当前有效”“最新变化”“草案 / 已生效”等时效判断。
- 主管机关、办理路径、材料要求、官方入口。
- 多法域比较表中的关键差异。
- 依赖二手来源或弱来源的判断。
- 实务操作建议中的关键步骤。

## 8. 不得引用的内容

- 只有文件名、知识库名、网页标题，但没有可定位原文片段。
- 只有域名标签，但没有具体页面、文件、法规 / 规则名、条号或章节。
- 搜索摘要不足以支撑结论的内容。
- 推测性常识补全。
- AI 生成答案本身。
- `[Valid]`、`[Current Date]`、`[X]`、`[有效]`、`[现行有效]`、无脚注实体的纯数字标记等伪引用。

## 9. 冲突处理

来源冲突时：

- 分别列明冲突来源。
- 比较效力层级、发布时间、适用范围和具体事实。
- 如果仍无法收口，标为“不明确”或“需当地确认”。

不要把冲突来源揉成一个看似确定的折中结论。

## 10. 翻译和原文

- 对非中文法域，关键术语尽量保留英文或当地官方语言原文。
- 用中文交付时，第一次出现原文术语后应解释含义。
- 不把机器翻译或非官方译文当作无风险原文。

## 11. 策略路径对比表

复杂路径选择问题可增加策略路径对比表：

| 路径 | 适用情形 | 可行性 | 法律依据 | 关键风险 | 证据强度 | 建议动作 |
| --- | --- | --- | --- | --- | --- | --- |

触发条件：

- 用户问题涉及“怎么做”“选哪条路径”“用什么结构”“是否直接设立 / SPV / 代理 / EOR”等。
- 至少存在两条可行路径。
- 不同路径的合规成本、审批、税务、劳动、责任承担或退出机制有实质差异。

不触发：

- 法条定位、案例定位、单点规则确认或只有唯一合法路径的问题。
- 缺少来源支撑的路径不得写成确定推荐，只能标为待验证。
